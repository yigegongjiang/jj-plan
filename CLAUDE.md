# jjplan agent rules

## 目标

本仓库用于开发 `jjplan`：一个极稳定的 Spec/Task 计划系统。

工程目标不是构建复杂项目管理工具，而是保证 Spec/Task 的创建、拆解、跟踪长期可靠。

稳定性优先级高于功能扩展、UI 质量、架构抽象和代码重构。

## 核心能力

- Spec：记录计划意图。
- Task：拆解 Spec。
- Status：跟踪 `todo`、`doing`、`blocked`、`done`。
- Completion：所有 Task 完成后，Spec 才能完成。

## 开发方式

- 把 CLI/API 行为视为稳定契约；数据结构是内部实现。
- 修改前先确认现有行为；修改后保持兼容。
- 数据模型按当前需求直接调整，保持简单一致。
- 新能力必须落在 Spec / Task 工作流内。
- 优先修正缺陷和边界条件，而不是扩展产品形态。
- 只做必要改动；直接实现优先于新抽象。

### For AI/LLM/Claude code /Codex

- 用户如果不明说不允许 push，那么可以参考 deploy.md 中的说明完成 push 和 tag 的创建（用户会自行通过 cloudflare url 完成访问验收)
- 版本号的修改尽量使用 3 位小版本调整。重大功能改造才允许修改 2 位版本号。
- 有时候是对上一个功能调整的不满意，这个时候应该使用 git amend，然后 force push 并打 tag。

## 自举

- `jjplan` CLI 已可实际使用。本工程既是它的产出物，也是它的使用者。
- AI 在维护本工程时，必须通过 `jjplan` 命令创建、拆解、跟踪本工程自身的 Spec/Task，而不是在对话或散落的 Markdown 中另起计划。
- 这种自举关系是检验稳定性的第一现场：用不顺即视为缺陷，优先修复。

## 边界

- 不追求复杂 UI；页面只需稳定、清晰、可用。
- 不增加 Spec / Task 之外的产品模型，除非用户明确要求。
- 不引入额外运行时、服务形态、存储系统或后台常驻进程。
- 不为数据层设计复杂兼容层、抽象层或迁移策略。
- 不为“更完整”而扩展范围；稳定、窄、可维护是默认答案。

## 文本内容要求

所有文本内容的查阅，不是人类，都是 AI/LLM。所以不需要兼容人类的阅读体验，能够让 AI/LLM 高质量理解即可。
